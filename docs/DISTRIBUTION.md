# Distribution Contract

`dist/` is tracked generated output used through raw GitHub, jsDelivr, or copies
inside consuming repositories. Edit source under `locales/` or `seo/`, then run
the owning builders.

## Translation Outputs

| Path | Inputs | Intended consumer |
| --- | --- | --- |
| `dist/cloud/{locale}.json` | cloud + modules + shared | Flyto2 Cloud |
| `dist/landing/{locale}.json` | landing + shared | Landing page |
| `dist/app/{locale}.json` | app + selected Cloud template + shared | Mobile app |
| `dist/code/{locale}.json` | code + shared | Flyto2 Code |
| `dist/console/{locale}.json` | console + shared | Console |
| `dist/data/{locale}.json` | data + shared | Data product |
| `dist/engine/{locale}.json` | engine + shared | Engine surfaces |
| `dist/{locale}.json` | all recognized projects | Full/admin compatibility consumers |

Each bundle includes locale metadata, a content-derived version, merged-file
and key counts, and nested `translations`. Every scope has `manifest.json` with
per-locale completion. `dist/locale-meta.json` contains shared locale metadata.
The builder synchronizes the hand-maintained root `manifest.json` coverage
fields from `dist/manifest.json`; freshness checks cover both files.

The `data` and `engine` output scopes currently contain shared fallback source
plus any future dedicated catalogs. Their named source directories do not yet
exist, so consumers must not infer dedicated product translation coverage from
the presence of generated bundles alone.

`dist/cortex/` is a tracked compatibility artifact but is not generated by the
current `SCOPES` mapping. Consumers must not assume it is fresh. Its migration
requires a named source scope and generator coverage; that work remains visible
in `STATE.md` and `tasks.md`.

## SEO Output

`dist/seo-manifest.json` is generated only by
`scripts/build-seo-manifest.py`. It includes a source hash so `--check` can
detect stale output without writing.

## Consumer Guarantees

- Identical recognized source content produces identical translation versions.
- Empty translated strings may remain and require consumer English fallback.
- A manifest completion percentage measures non-empty keys after merge.
- CDN cache invalidation is best effort; pinned revisions are preferred for
  reproducible releases.
- Consumers validate response shape and retain a bundled English fallback.

## Regeneration

```bash
python3 scripts/validate.py --strict
python3 scripts/build-dist.py
python3 scripts/build-seo-manifest.py
git diff -- dist/
```
